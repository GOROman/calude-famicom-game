# Step 3: ブロックの上に立つということ

ブロックは、乗れて初めてブロックになる。それまでは背景の模様にすぎない。Step 3 は、模様を物体に変える工程だった。

## 点で当たる

当たり判定というと矩形同士の重なりを思い浮かべるけれど、ファミコンの計算力で毎フレームAABBをやるのは贅沢だ。そこで点判定にした。プレイヤーの輪郭から代表点を選び、その点が「ソリッドの中にいるか」だけを調べる。

横方向は前縁の3点(頭・腰・足)。移動してみて、どれかがブロックに食い込んだら移動をなかったことにする。2px/フレームの移動速度と16px格子の組み合わせだと、この「取り消すだけ」でぴったり壁に張り付く。数のマジックだ。

縦方向は足元の2点と頭上の2点。落下中に足元がソリッドに入ったら、その面の上端にY座標をスナップして着地。上昇中に頭が入ったら、ブロックの下面で速度をゼロにする。マリオが天井に頭をぶつけて「ゴツン」と落ちる、あの感触がそのまま出た。

一番気に入っているのは「接地中に足元が空になったら落下開始」の3行だ。これだけで、ブロックの端から歩いて踏み外せるようになる。落ちる自由も、物理のうちだ。

## ブロックのサイズはマリオに合わせる

このタイミングでブロックを16x16に変えた。それまでの8x8タイル模様は、スーパーマリオの世界の「1ブロック」の半分しかなかった。レベルデータを16px単位のメタ列64個に組み直し、輪郭とハイライト付きの「1個のブロック」として描き直す。当たり判定の格子と見た目の格子が一致した瞬間、ゲームが急にゲームらしくなった。

## 影の話

実はこの回、接地感を出すために影も実装した。プレイヤーの真下の面に楕円の影が落ち、ジャンプすると高度に応じて小さくなる。テストも書いて、ちゃんと動いた。

そして次の日、「影は不要」の一言で消えた。

ゲーム開発とはそういうものだ。作った機能の何割かは捨てられる。でも影のために書いた「真下の面を求める」コードの考え方は、のちの敵の接地処理に生きている。消えた機能は、消えたコードとイコールではない。
